## 1. リサーチ（NotebookLM）

`notebooklm` skillを使う。手順：

1. `notebooklm auth check --test --json` で認証確認（切れていたら `notebooklm login`）
2. `notebooklm create "<テーマ> 長期運用ブログ リサーチ" --json` でノートブック作成 → `notebooklm use <id>`
3. トピックを5〜7個の検索クエリに分解し、`notebooklm source add-research "<query>" --mode fast --import-all --json` を1つずつ実行
   - 例（通貨ペア系記事の場合）: 過去の為替推移、現在の金融政策、ドル円・米ドルの将来予測、FX会社のスワップ比較、リスク要因（過去の暴落事例）、口座開設・購入手順
   - `--mode fast` で十分（`deep`は15〜30分かかるため通常は不要）
4. 収集後、`notebooklm ask "<具体的な質問>" --json` でテーマごとに要点を抽出（数値・引用元付きで返ってくる）
   - 質問は「〜を時系列で要約し、具体的な数値も含めてください」のように具体性を持たせる
   - 予測系は「複数の金融機関の予測をシナリオ別（中立/強気/弱気）に整理して」のように依頼すると使いやすい

### NotebookLMの役割について（2026-07-06 検証済み）

`notebooklm generate report --format blog-post` に記事執筆そのものを任せられないか検証した結果、**現行方針（NotebookLMは引用付きリサーチに徹し、記事の執筆・構成はClaudeが行う）を継続する**ことに決定。理由：

- **ハルシネーションを検知**: 同一ノートブック・同一ソースから英語版と日本語版を生成したところ、次期FRB議長として名指しされた人物が**英語版ではKevin Hassett、日本語版ではKevin Warsh**と、生成のたびに矛盾する「事実」が出力された。`generate report`には引用元（出典）が一切表示されないため、どの記述が実ソースに基づくものかを検証する手段がない。
- `ask`コマンド（本ワークフローで採用している方式）は毎回引用元付きで返るため、この検証ができる——これが`ask`ベースのリサーチ方式を維持する決め手。
- その他の理由: `generate report`の出力は記事の核となる要素（FX会社・スワップ比較、シミュレーション、口座開設手順、免責文、挨拶文などブログ固有のフォーマット）を満たさず、分量も目標に届かず、文体もブログの口調と異なる。

→ 今後も`generate report --format blog-post`を試す場合は、出力を鵜呑みにせず`ask`で個別に事実確認してから使うこと。

## 2. 執筆ルール

Obsidianを使う場合、Templaterプラグイン導入後は `ai_workspace/templates/rational-lab_記事テンプレート.md` を新規記事作成時のテンプレートに指定すると、frontmatter（status/tags/date等）と章立ての雛形が自動挿入される。

- ボリューム目安: 1本8,000〜12,000文字程度の超長文記事（アメブロの長文記事として成立する分量）
- 構成: はじめに → 過去 → 現在 → 未来（複数シナリオ）→ 業者・商品比較 → シミュレーション（該当する場合）→ リスクと注意点 → 始め方ステップガイド → まとめ
- 投資・金融系の内容は必ず以下を守る：
  - 「安全に稼げる」という煽り一辺倒にせず、リスク（暴落事例・レバレッジの危険性など）を対で書く
  - 断定的な投資助言（「今買うべき」等）は避け、情報提供目的である旨を明記
  - 記事末尾に定型の免責文を入れる：
    > ※本記事は情報提供を目的としたものであり、特定の投資商品の売買を推奨するものではありません。◯◯取引には元本を上回る損失が生じるリスクがあります。投資判断は、必ずご自身の責任において行ってください。
- 文字数がすでに多く、新しい章や図解を追加する余地がない場合 → 既存の文章（特に個別の細かい説明）を要約して圧縮し、図解に語らせる形にする（数値の重複説明を削る）

## 3. 図解・グラフ生成（dataviz skill）

必ず `dataviz` skillを読み込んでから作業する。手順：

1. `references/palette.md` の配色を使う（カテゴリカル: blue `#2a78d6` → aqua `#1baf7a` → yellow `#eda100` → green `#008300` → violet `#4a3aa7`、サーフェス `#fcfcfb`、インク `#0b0b0b`/`#52514e`/`#898781`）
2. matplotlib + 日本語フォント（`C:\Windows\Fonts\YuGothM.ttc` 等をfontManagerに追加）で生成。`uv run --with matplotlib <script>.py` で実行（インストール不要、その場で解決される）
3. **1軸の原則を守る**: 桁違いの系列（例: レバレッジ25倍 vs 1倍のような複利シミュレーション）は対数軸(`ax.set_yscale('log')`)にする。線形のまま描画すると1本だけ突き抜けて他が潰れる（実際に一度失敗している）
4. 直接ラベル（系列の末尾に数値を添える）を基本とし、ラベル同士が重なる場合は手動でオフセットを調整する
5. 出力は `G:\マイドライブ\AIの作業場\blog\アメブロブログ記事\images\` に保存
6. 生成する主なグラフパターン：
   - 過去の推移＋主要ショック局面（暴落イベント）を赤マーカーで強調した折れ線グラフ
   - レバレッジ/条件別の比較（小規模棒グラフの並び、small multiples）
   - 複利シミュレーション（対数軸の折れ線、極端なケースには警告注釈を入れる）
7. 国旗など装飾画像も同じ palette/簡易図形で作れるが、**対象国・通貨の正式な国旗が既存の画像ライブラリにある場合はそちらを優先**（自作の簡易版より正確）。過去に自作の簡易メキシコ国旗を使って「変」と指摘されたため、実写・正式なものがあれば必ずそちらを使う。
   - **入手方法（2026-07-10確立）**: `https://flagcdn.com/w1280/<ISO 3166-1 alpha-2国コード小文字>.png`（例: 南アフリカ→`za`）から`curl`で直接ダウンロードする。国旗CDNとして広く使われている信頼できるパブリックソースで、簡易図形の自作より正確。ダウンロード後は記事の`images\`フォルダに`flag_<国名>.png`として保存し、通常の画像アップロード手順でAmebaにアップロードしてURLを取得する。挨拶文（「こんにちわ！rational-labです！」）の直後、`## はじめに`より前に中央揃えで配置する。
   - **「なびく布」風の演出（2026-07-10確立、1回失敗して修正済み）**: メキシコペソ記事の国旗画像（写真風の3D立体・布の質感でなびいている）と同じ「かっこよさ」を求められた場合、flagcdn.comのフラットな画像そのままだと物足りない。`uv run --with pillow --with numpy <script>.py`でnumpy+PILによる後処理を行うが、**実際のメキシコ国旗画像を取得して観察した結果、色の境界線（ストライプ）自体はほぼ真っ直ぐなまま**で、「なびき感」はほぼ100%シェーディング（陰影）だけで表現されていることが分かった。
     - **失敗した最初のアプローチ**: 複数のsin波を重ねた変位マップで色の境界線自体を波打つ形に大きく変形させたところ、南アフリカ国旗のようなY字/山形の複雑な図柄では境界線が不自然にねじれ「変」と指摘された。ポールから離れるほど振幅が増す「風になびく」タイプの変位は、直線的な三色旗（メキシコ）では目立たないが、対角線を含む複雑な図柄では破綻しやすい。
     - **正しいアプローチ**: (1)ジオメトリの変位はごく小さく留める（数px程度、色の境界線の形はほぼ保つ）、(2)緩やかな複数のsin波を重ねた「シェーディングマップ」を作り、`ImageFilter.GaussianBlur`で強くぼかして滑らかな明暗の帯（ひだ）にする、(3)元画像に明暗マップを掛け合わせる（乗算）、(4)細かいノイズで布目のテクスチャを薄く乗せる、(5)軽いヴィネットを足す。旗の絵柄そのものは歪ませず、光と影だけで立体感を出すのがコツ。
     - フラット画像は上書きせず`flag_<国名>_wave.png`など別名で保存し、なびき加工後のファイルを記事に使う。
     - **「高級な布感」を出す追加調整（2026-07-10）**: 上記だけだと少し安っぽく見える場合、(a)ひだの明暗コントラストを広げる（例: 0.62〜1.36倍程度）、(b)緩やかな対角線状の「サテンの光沢」帯を追加する——`sin(u*係数 - v*係数)`を0〜1に正規化してから3乗程度でべき乗し帯を細く鋭くし、強めにガウスぼかしをかけてから元画像に加算、(c)ノイズだけでなく`sin(x+y)`と`sin(x-y)`を組み合わせた綾織り風の微細パターンを重ねる、(d)`ImageEnhance.Color`で彩度を1.15〜1.2倍、`ImageEnhance.Contrast`で1.05〜1.1倍程度上げる——の4点で高級感が大きく向上する。
     - **ユーザー自身が国旗の実写画像を持っている場合はそちらを最優先（2026-07-10）**: 上記の加工をどれだけ工夫しても、実際に布を撮影した商用ストック写真の質感には及ばない。ユーザーがデスクトップ等のフォルダに実写の国旗画像（ピンやグロメット、縫い目まで写った本物の布の写真）を持っている場合は、そのファイルパスを教えてもらい、白背景の余白を`numpy`で非白色ピクセルのbounding boxを検出してトリミングするだけで十分に使える。合成で作り込むより先に「持っている画像はないか」を確認するのが早い。ユーザーが会話にファイルを直接貼り付けた場合、そのファイルがどこに保存されているか（一時ファイルとして自動保存されないことがある）を明示的に尋ねる必要がある。

## 3.5 ファクトチェック（投稿前の数値検証、2026-07-10追加・必須ステップ）

執筆・図解が終わったら、**アメブロに下書き保存する前に必ず**この記事本文中の数値をリサーチ段階の引用データと照合する。手順：

1. 記事本文から数値を含む主張を全て抽出する（％、円、bp、倍、年、人数など）。複利シミュレーションのように自分で計算した数値（前提を明示した試算）は対象外——照合すべきは「外部の事実」としてリサーチした数値のみ。
2. この記事のリサーチで使った`notebooklm ask`の各回答は引用元（`references`のsource_id・cited_text）付きで返っているので、抽出した数値ごとに「どの引用のどの回答から来た数値か」を照合する。
   - **新規に`ask`を追加で叩き直す必要はない**——同一セッション内で既に取得済みの引用データと突き合わせるだけ（高速・推奨）。
   - 引用元が明確に見つかる数値 → ✅照合済み
   - 引用元が見つからない・別記事からの使い回し・一般論として書いただけの数値 → ⚠️要確認（この記事のリサーチでは検証していない旨を明記）
3. 結果を`<記事slug>_factcheck.md`として記事と同じフォルダに保存する（Dataviewで後から一覧・監査できるように）。フォーマット：

   ```markdown
   | 数値 | 記事内の文脈 | 判定 | 根拠 |
   |---|---|---|---|
   | 7.00% | SARB政策金利 | ✅照合済み | ask「南アフリカ準備銀行(SARB)の現在の政策金利...」の回答、citation[2][3] |
   | 300%以上 | 証拠金維持率の目安 | ⚠️要確認 | この記事のリサーチでは未検証。過去記事(メキシコペソ)の一般的な目安を流用 |
   ```

4. ⚠️要確認が1件でもある場合は、**下書き保存の前にユーザーに報告し、修正するか・注記を入れるか・そのまま進めるかを確認する**。ユーザーの回答を待たずに勝手に下書き保存まで進めない。
5. 全て✅、またはユーザーが⚠️を許容した場合のみ、次の「4. アメブロ投稿の自動化」に進む。

**この仕組みの限界**：あくまで「この記事の執筆時にリサーチした引用と一致するか」の照合であり、引用元サイト自体の正確性までは検証しない（一次資料の誤りは検出できない）。また抽出・照合はClaudeが記事ごとに手動で行うため、完全に自動化されたスクリプトではなく実行を忘れないための手順として運用する。

## 4. アメブロ投稿の自動化

詳細な技術手順は別メモリ `ameba_blog_automation.md`（Claude内部の記憶ファイル）に保存済み。要点のみ再掲：

- 必ず**完全に分離した専用Chromiumインスタンス**で操作する（Playwrightがダウンロード済みの `chromium-*/chrome-win64/chrome.exe` を新規 `--user-data-dir` + `--remote-debugging-port` で起動）。メインの普段使いのChromeに誤接続しないよう、起動直後に `contexts[0].pages` が1枚の `about:blank` だけであることを確認してから進める。
- ログインはユーザー本人に行ってもらう（認証情報は代行入力しない）
- 新規投稿: `https://blog.ameba.jp/ucs/entry/srventryinsertinput.do`
- 記事管理（削除はここだけ）: `https://blog.ameba.jp/ucs/entry/srventrylist.do?entry_ym=YYYYMM`
- 本文HTMLは `document.querySelector('.CodeMirror').CodeMirror.setValue(html); cm.save();` で流し込む（生の `textarea.value=` では反映されない）
- 画像は「写真・動画」→「アップロード」→ `page.expect_file_chooser()` でローカルファイルを直接アップロードし、`li` 内の `div[data-image]` 属性から実URL（`stat.ameba.jp/user_images/...`）を取得して `<img src>` に直接埋め込む（プレビューに手動で挿入する必要はない）
- **必ず「下書き保存」までで止める。「投稿する」は絶対に押さない**——公開は必ずユーザー自身に行ってもらう
- 削除は下書き一覧のドロップダウンではなく `srventrylist.do` の記事一覧からのみ可能（`削除` → 確認モーダルで「削除する」）
- **同じ`--user-data-dir`のプロファイルが既にウィンドウとして開いている状態で別ポート番号で起動し直しても、既存インスタンスにウィンドウが追加されるだけで新ポートではCDP待受されない**（`connect_over_cdp`が`ECONNREFUSED`になる）。この場合はさらに別のポート番号で起動し直す（大抵はロックが外れて正しく待受される）か、既存のchrome.exeを`taskkill`で終了してから起動し直す

### 2026-07-10 追記：投稿画面が新エディタ（CodeMirrorなし）に変わっていた場合

Amebaの新規投稿画面がリニューアルされ、従来の「HTML表示」トグル＋CodeMirrorの構成ではなく、右上「他のエディタを使う」→設定・管理の「記事投稿画面」で**タグ編集エディタ**に切り替える方式になっていることがある（2026-07-10時点で確認）。

- 切り替え手順：新規投稿画面の右上「他のエディタを使う」→`設定・管理`ページの「記事投稿画面」で**タグ編集エディタ**を選択→ページ下部の保存ボタンで確定
- タグ編集エディタは本文が生の`<textarea id="entryTextArea" name="entry_text">`（CodeMirrorではない）なので、`locator.fill(html_string)`で直接流し込める。タイトルは`input[name="entry_title"]`
- 画像アップロード後の`data-image`属性からのURL取得手順は変わらず有効（`ul.p-images-imageList__list li div[data-image]`）。ただし新規アップロードのthumbnail表示順は「新しい順」なので、複数枚を一括アップロードした場合は幅・高さの組み合わせで区別し、アップロードIDの大小（同日同時刻なら末尾の連番）で時系列を判定するとよい
- 「アップロード」ボタンはテキストロケータで直接クリックすると別要素（screen-reader-only文言）に当たることがあるため、`nth()`で正しい候補を選び、`bounding_box()`を取得してから`mouse.click(cx, cy)`する方が確実
- 下書き保存ボタンをクリックしても同一URL（`srventryinsertinput.do`）に留まる（別ページへ遷移しない）。保存確認は`srventrylist.do?entry_ym=YYYYMM`に遷移してタイトルと「下書き」ステータス文言があるかで確認する
