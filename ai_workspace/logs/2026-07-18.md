## メモ・気づき

- Netlifyの新しい「クレジット」課金体系（AI usage / agent runner credit等の文言もアカウント情報に見える）は、単純な「デプロイ回数上限」ではなく複数リソースを束ねた共通通貨。今後Netlifyの利用制限まわりで質問されたら、`netlify api listAccountsForUser`で`plan_credits`・`current_billing_period_start`・`next_billing_period_start`・`grace_topup_granted_at`などを直接確認するのが早い。
- 今回の学びとして、同一セッション内で細かい修正を繰り返すたびに毎回デプロイするのはクレジット消費の面でも非効率だった。今後は複数の小改修をローカルで積み重ねてから最後に1回デプロイする方針に統一。
- 「フリーズしたヒーロー数値」設計（v1改22）は詳細設定側の試行錯誤には有効だが、プロフィール欄という「前提条件そのものの変更」にまで適用してしまうと、入力を変えても数字が動かないという致命的なバグに見える。今後同様の「一度計算したら固定」機能を作る際は、どの操作が「前提の変更（リセットすべき）」でどの操作が「試行錯誤（維持すべき）」かを明確に線引きすること。
- 単一HTMLファイルの動作確認は、Netlifyデプロイよりも先にClaude Artifactを使う方針に統一。外部リソース依存のないツールなら、ラッパータグを外してそのまま公開でき、クレジット消費もなく即座にURLを渡せる。
- Artifact抽出の行番号バグは、「一度計算したファイルの構造情報（行番号）を次回もそのまま使い回す」という横着が原因だった。ソースファイルは毎回変わる前提で、抽出のたびに動的に再計算するのが鉄則。
- 「デフォルト値が意図せず楽観的な前提になっている」系のバグ（アルバイト継続年数の初期値＝想定寿命、受給開始年齢セレクトの初期値＝60歳など）が今日だけで複数回見つかった。UIの見た目だけでなく、実際にpage.evaluateや`input_value()`でDOM値を検証しないと気づけないタイプのバグなので、今後も「デフォルト状態」を明示的にテストする習慣を継続する。
- ユーザーが公式資料（ねんきんネットのスクリーンショット）を見せてくれたことで、簡易計算だった手取り試算を厚労省公式の計算式にそのまま置き換えられた。ユーザー提供の一次情報は精度向上の良いきっかけになる。
