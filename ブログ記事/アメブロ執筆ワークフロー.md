## 1. リサーチ（NotebookLM）

`notebooklm` skillを使う。手順：

1. `notebooklm auth check --test --json` で認証確認（切れていたら `notebooklm login`）
2. `notebooklm create "<テーマ> 長期運用ブログ リサーチ" --json` でノートブック作成 → `notebooklm use <id>`
3. トピックを5〜7個の検索クエリに分解し、`notebooklm source add-research "<query>" --mode fast --import-all --json` を1つずつ実行
   - 例（通貨ペア系記事の場合）: 過去の為替推移、現在の金融政策、ドル円・米ドルの将来予測、FX会社のスワップ比較、リスク要因（過去の暴落事例）、口座開設・購入手順
   - `--mode fast` で十分（`deep`は15〜30分かかるため通常は不要）
4. 収集後、`notebooklm ask "<具体的な質問>" --json` でテーマごとに要点を抽出（数値・引用元付きで返ってくる）
   - 質問は「〜を時系列で要約し、具体的な数値も含めてください」のように具体性を持たせる
   - 予測系は「複数の金融機関の予測をシナリオ別（中立/強気/弱気）に整理して」のように依頼すると使いやすい

### NotebookLMの役割について（2026-07-06 検証済み）

`notebooklm generate report --format blog-post` に記事執筆そのものを任せられないか検証した結果、**現行方針（NotebookLMは引用付きリサーチに徹し、記事の執筆・構成はClaudeが行う）を継続する**ことに決定。理由：

- **ハルシネーションを検知**: 同一ノートブック・同一ソースから英語版と日本語版を生成したところ、次期FRB議長として名指しされた人物が**英語版ではKevin Hassett、日本語版ではKevin Warsh**と、生成のたびに矛盾する「事実」が出力された。`generate report`には引用元（出典）が一切表示されないため、どの記述が実ソースに基づくものかを検証する手段がない。
- `ask`コマンド（本ワークフローで採用している方式）は毎回引用元付きで返るため、この検証ができる——これが`ask`ベースのリサーチ方式を維持する決め手。
- その他の理由: `generate report`の出力は記事の核となる要素（FX会社・スワップ比較、シミュレーション、口座開設手順、免責文、挨拶文などブログ固有のフォーマット）を満たさず、分量も目標に届かず、文体もブログの口調と異なる。

→ 今後も`generate report --format blog-post`を試す場合は、出力を鵜呑みにせず`ask`で個別に事実確認してから使うこと。

## 2. 執筆ルール

Obsidianを使う場合、Templaterプラグイン導入後は `ai_workspace/templates/rational-lab_記事テンプレート.md` を新規記事作成時のテンプレートに指定すると、frontmatter（status/tags/date等）と章立ての雛形が自動挿入される。

- ボリューム目安: 1本8,000〜12,000文字程度の超長文記事（アメブロの長文記事として成立する分量）
- 構成: はじめに → 過去 → 現在 → 未来（複数シナリオ）→ 業者・商品比較 → シミュレーション（該当する場合）→ リスクと注意点 → 始め方ステップガイド → まとめ
- 投資・金融系の内容は必ず以下を守る：
  - 「安全に稼げる」という煽り一辺倒にせず、リスク（暴落事例・レバレッジの危険性など）を対で書く
  - 断定的な投資助言（「今買うべき」等）は避け、情報提供目的である旨を明記
  - 記事末尾に定型の免責文を入れる：
    > ※本記事は情報提供を目的としたものであり、特定の投資商品の売買を推奨するものではありません。◯◯取引には元本を上回る損失が生じるリスクがあります。投資判断は、必ずご自身の責任において行ってください。
- 文字数がすでに多く、新しい章や図解を追加する余地がない場合 → 既存の文章（特に個別の細かい説明）を要約して圧縮し、図解に語らせる形にする（数値の重複説明を削る）

## 3. 図解・グラフ生成（dataviz skill）

必ず `dataviz` skillを読み込んでから作業する。手順：

1. `references/palette.md` の配色を使う（カテゴリカル: blue `#2a78d6` → aqua `#1baf7a` → yellow `#eda100` → green `#008300` → violet `#4a3aa7`、サーフェス `#fcfcfb`、インク `#0b0b0b`/`#52514e`/`#898781`）
2. matplotlib + 日本語フォント（`C:\Windows\Fonts\YuGothM.ttc` 等をfontManagerに追加）で生成。`uv run --with matplotlib <script>.py` で実行（インストール不要、その場で解決される）
3. **1軸の原則を守る**: 桁違いの系列（例: レバレッジ25倍 vs 1倍のような複利シミュレーション）は対数軸(`ax.set_yscale('log')`)にする。線形のまま描画すると1本だけ突き抜けて他が潰れる（実際に一度失敗している）
4. 直接ラベル（系列の末尾に数値を添える）を基本とし、ラベル同士が重なる場合は手動でオフセットを調整する
5. 出力は `G:\マイドライブ\AIの作業場\ブログ記事\images\` に保存
6. 生成する主なグラフパターン：
   - 過去の推移＋主要ショック局面（暴落イベント）を赤マーカーで強調した折れ線グラフ
   - レバレッジ/条件別の比較（小規模棒グラフの並び、small multiples）
   - 複利シミュレーション（対数軸の折れ線、極端なケースには警告注釈を入れる）
7. 国旗など装飾画像も同じ palette/簡易図形で作れるが、**対象国・通貨の正式な国旗が既存の画像ライブラリにある場合はそちらを優先**（自作の簡易版より正確）。過去に自作の簡易メキシコ国旗を使って「変」と指摘されたため、実写・正式なものがあれば必ずそちらを使う。

## 4. アメブロ投稿の自動化

詳細な技術手順は別メモリ `ameba_blog_automation.md`（Claude内部の記憶ファイル）に保存済み。要点のみ再掲：

- 必ず**完全に分離した専用Chromiumインスタンス**で操作する（Playwrightがダウンロード済みの `chromium-*/chrome-win64/chrome.exe` を新規 `--user-data-dir` + `--remote-debugging-port` で起動）。メインの普段使いのChromeに誤接続しないよう、起動直後に `contexts[0].pages` が1枚の `about:blank` だけであることを確認してから進める。
- ログインはユーザー本人に行ってもらう（認証情報は代行入力しない）
- 新規投稿: `https://blog.ameba.jp/ucs/entry/srventryinsertinput.do`
- 記事管理（削除はここだけ）: `https://blog.ameba.jp/ucs/entry/srventrylist.do?entry_ym=YYYYMM`
- 本文HTMLは `document.querySelector('.CodeMirror').CodeMirror.setValue(html); cm.save();` で流し込む（生の `textarea.value=` では反映されない）
- 画像は「写真・動画」→「アップロード」→ `page.expect_file_chooser()` でローカルファイルを直接アップロードし、`li` 内の `div[data-image]` 属性から実URL（`stat.ameba.jp/user_images/...`）を取得して `<img src>` に直接埋め込む（プレビューに手動で挿入する必要はない）
- **必ず「下書き保存」までで止める。「投稿する」は絶対に押さない**——公開は必ずユーザー自身に行ってもらう
- 削除は下書き一覧のドロップダウンではなく `srventrylist.do` の記事一覧からのみ可能（`削除` → 確認モーダルで「削除する」）
